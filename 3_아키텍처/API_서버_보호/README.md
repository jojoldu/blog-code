# Public API는 서버까지 오지 않게 하기

이전에 [Public API의 CDN 캐시](https://jojoldu.tistory.com/779)에 대해 글을 쓴 적이 있다.

인프런 상단 헤더에 노출되는 강의 카테고리처럼 로그인 여부와 관계없이 모든 사용자에게 동일한 응답을 주는 API가 있었다.
이 API는 모든 페이지에서 호출되다보니 페이지 조회수만큼 애플리케이션 서버 요청이 발생했다.

처음에는 자연스럽게 Redis 캐시나 로컬 캐시를 떠올렸다.
하지만 Redis 캐시와 로컬 캐시는 DB 요청을 줄일 뿐, 애플리케이션 서버로 들어오는 요청수 자체를 줄이지는 못한다.

```text
Client -> API Server -> Local Cache
Client -> API Server -> Redis
```

그래서 해당 API 응답을 CDN에서 캐시하도록 변경했다.

```text
Client -> CDN -> API Server
```

동일한 JSON 응답이 CDN에서 끝나니 애플리케이션 서버까지 오지 않게 되었다.
그 결과 해당 public API의 요청수와 로드가 크게 줄었다.

이 개선을 하고 나니 public API를 볼 때 이전에는 잘 보이지 않던 지표가 보이기 시작했다.

응답 시간이 아니라 원본 서버까지 들어온 요청수였다.

Public API는 외부에 열려 있다.
로그인하지 않은 사용자도 호출할 수 있고, 검색 봇도 호출할 수 있고, 브라우저도 호출할 수 있고, 누군가 작성한 스크립트도 호출할 수 있다.
서비스 안쪽의 API보다 호출 주체를 통제하기 어렵다.

Public API를 애플리케이션 서버에서 빠르게 처리하는 것만으로는 부족할 때가 있다.
호출 주체를 통제하기 어렵다면, 애플리케이션 서버가 직접 맞는 요청수를 줄이는 쪽까지 같이 봐야 한다.

## Public API는 호출자가 불명확하다

로그인 API나 마이페이지 API처럼 사용자가 명확한 API는 그래도 어느 정도 호출 주체를 식별할 수 있다.
세션이 있고, 사용자 ID가 있고, 조직 ID가 있고, 권한이 있다.

반면 public API는 다르다.

강의 카테고리, 공개 강의 목록, 공개 검색, 공개 프로필, 공개 아티클, SEO용 데이터처럼 로그인하지 않아도 접근 가능한 API는 호출자가 명확하지 않다.
물론 IP, User-Agent, Referer 정도는 볼 수 있지만 이것만으로는 충분하지 않다.

정상 사용자가 브라우저로 접근한 것인지,
검색 엔진 봇인지,
가격 정보를 긁어가는 스크래퍼인지,
잘못 만든 프론트엔드 코드가 같은 요청을 반복하는 것인지,
단순히 캐시가 깨져서 모든 요청이 원본 서버로 몰리는 것인지 구분하기 어렵다.

이런 요청을 모두 애플리케이션 서버가 직접 받아서 처리하면 서버가 맡는 일이 많아진다.

비즈니스 데이터를 조회하고 응답하는 것뿐만 아니라,
반복 호출을 견디고,
비정상 호출을 구분하고,
비용이 큰 요청을 제한하고,
캐시 가능 여부를 판단하고,
로그와 지표까지 남겨야 한다.

물론 처음에는 이렇게 시작할 수 있다.
작은 서비스에서 public API 하나를 위해 CDN, Gateway, WAF, Rate Limit 정책까지 모두 준비하는 것은 과할 수 있다.

다만 public API의 호출량이 서비스 전체 요청에서 큰 비중을 차지하기 시작하면 컨트롤러 코드만 봐서는 부족해진다.
이 API가 어느 계층까지 내려오는지, 어디서 캐시되고, 어디서 제한되고, 어떤 지표로 관측되는지를 같이 봐야 한다.

## 서버 앞에서 끝낼 수 있는 응답

Public API 중에는 모든 사용자에게 동일한 응답을 주는 것들이 있다.

- 강의 카테고리
- 공개 태그 목록
- 공개 배너 설정
- 공통 코드
- 공개 랭킹
- 비로그인 상태에서도 동일한 메타 데이터

이런 API는 CDN 캐시 후보가 된다.

이미 Redis 캐시나 로컬 캐시로 충분히 빠른 API도 있다.
그런데 서버가 매번 요청을 받아서 캐시에서 값을 꺼내고 있다면, 원본 서버 요청수는 줄지 않는다.

응답이 모든 사용자에게 동일하고,
권한에 따라 달라지지 않고,
쿠키나 세션에 의존하지 않고,
일정 시간 정도 stale한 응답을 허용할 수 있다면 CDN에서 끝낼 수 있다.

```text
Client -> CDN Cache
```

이 구조에서는 애플리케이션 서버가 관여하지 않는다.
요청이 빠르게 처리되는 수준을 넘어, 서버 입장에서는 아예 받지 않은 요청이 된다.

다만 CDN 캐시는 조심해야 한다.

Public API라고 해서 모두 CDN에 올릴 수 있는 것은 아니다.
로그인 여부에 따라 응답이 달라지는 API, 사용자 권한에 따라 일부 필드가 달라지는 API, 쿠키 기반으로 실험군이 달라지는 API, 지역이나 언어에 따라 응답이 달라지는 API는 캐시 키를 잘못 잡으면 문제가 된다.

특히 쿠키와 인증 헤더는 조심해야 한다.
응답이 사용자별로 달라질 수 있는데 CDN이 이를 구분하지 못하면 다른 사용자의 응답이 섞일 수 있다.

그래서 CDN 캐시 대상이 되는 public API는 가능하면 팀 안에서 식별 가능해야 한다.

예를 들어 다음과 같은 식이다.

```text
/api/v1/public/categories
/api/v1/public/tags
/api/v1/public/banners
```

URI에 `public`을 넣어야만 한다는 뜻은 아니다.
별도 도메인을 쓸 수도 있고, 라우팅 그룹이나 API 문서 태그로 구분할 수도 있다.
어떤 방식을 쓰든 팀 안에서 "이 API는 사용자와 무관하게 동일한 public 응답이다"라는 신호가 명확해야 한다.

이 신호가 있어야 CDN 캐시 정책, 인증 제외 정책, Rate Limit 정책, 로그 정책을 별도로 가져갈 수 있다.

## Public API와 Private API는 다르게 다뤄야 한다

HTTP API를 설계할 때 URI는 리소스를 중심으로 잡는 것이 좋다.
이 원칙은 여전히 중요하다.

다만 운영 정책까지 생각하면 public API와 private API는 분리해서 볼 필요가 있다.

둘은 호출 주체도 다르고,
캐시 가능성도 다르고,
Rate Limit 기준도 다르고,
로그에서 봐야 할 지표도 다르다.

예를 들어 강의 목록 API가 있다고 해보자.

비로그인 사용자에게 노출되는 공개 강의 목록과,
관리자가 보는 강의 목록은 이름은 비슷해도 운영 특성이 완전히 다르다.

공개 강의 목록은 트래픽이 많고 캐시 가능성이 높다.
관리자 강의 목록은 트래픽은 적지만 권한이 중요하고, 사용자별 응답이 달라질 수 있다.

이 둘을 같은 API로 두고 내부에서 권한이나 파라미터에 따라 응답을 바꾸면 처음에는 편하다.
하지만 시간이 지나면 운영 정책을 분리하기 어렵다.

공개 API에는 CDN 캐시를 적용하고 싶은데 관리자 응답이 섞여 있으면 어렵다.
공개 API에는 더 강한 Rate Limit을 두고 싶은데 내부 관리자 API까지 같이 영향을 받으면 곤란하다.
공개 API의 트래픽 지표를 따로 보고 싶은데 같은 URI에 섞여 있으면 분석하기 어렵다.

그래서 public API는 응답 책임을 명확하게 가져가는 것이 좋다.

```text
GET /api/v1/public/courses
GET /api/v1/admin/courses
GET /api/v1/my/courses
```

위 URI가 항상 더 낫다는 뜻은 아니다.
기존 API 컨벤션에 따라 다른 방식으로 나눌 수 있다.
별도 도메인, 별도 라우터, 별도 API 태그로 구분해도 된다.

Public API와 Private API를 한 엔드포인트 안에서 조건문으로 계속 갈라내기 시작하면, 나중에 운영 정책을 분리하기가 어려워진다.

## Public API의 Rate Limit

Public API는 인증되지 않은 사용자가 호출할 수 있기 때문에 사용자 ID 기준으로 제한하기 어렵다.

그래서 보통 다음 기준을 조합하게 된다.

- IP
- User-Agent
- Referer
- 국가나 ASN
- API Key가 있다면 API Key
- URI

이 기준들은 완벽하지 않다.
IP는 NAT나 프록시 뒤에서 여러 사용자가 공유할 수 있고, User-Agent와 Referer는 조작될 수 있다.
국가나 ASN 기준도 정상 사용자를 같이 막을 수 있다.

그럼에도 아무 제한이 없는 것보다는 낫다.

특히 public API는 비용이 큰 API와 저렴한 API를 구분해야 한다.

카테고리 목록처럼 CDN에서 대부분 끝나는 API는 원본 서버 기준으로는 낮은 요청량만 허용해도 된다.
정상 상황이라면 CDN 캐시가 대부분 처리하기 때문이다.

반면 검색 API나 필터링 API처럼 쿼리 조합이 많아 캐시 효율이 낮고 DB나 검색 엔진 비용이 큰 API는 더 보수적으로 제한해야 한다.

```text
GET /api/v1/public/categories     -> CDN 캐시 + 원본 낮은 제한
GET /api/v1/public/courses        -> CDN 또는 짧은 TTL + 일반 제한
GET /api/v1/public/search         -> 캐시 제한적 + 강한 제한
GET /api/v1/public/autocomplete   -> 짧은 TTL + IP/URI 기준 제한
```

전체 public API에 같은 제한을 두면 비용 차이가 드러나지 않는다.

Public API라는 공통점은 있지만 각 API가 만드는 부하는 다르다.
캐시 가능성도 다르고, 사용자 경험에 미치는 영향도 다르다.

자동완성 API는 사용자가 한 글자 입력할 때마다 호출될 수 있다.
그래서 너무 강하게 막으면 검색 경험이 나빠진다.
대신 짧은 TTL 캐시나 클라이언트 디바운스와 함께 설계해야 한다.

검색 API는 사용자가 명시적으로 실행하는 요청이다.
자동완성보다 호출 빈도는 낮지만 비용은 더 클 수 있다.
그래서 페이지네이션, 최대 조회 범위, 정렬 조건, 필터 조합 제한까지 같이 봐야 한다.

Rate Limit을 정할 때는 숫자보다 먼저 API의 비용 구조와 사용 패턴을 봐야 한다.

## 캐시 실패를 전제로 보기

CDN 캐시를 적용했다고 해서 안심하면 안 된다.

캐시는 언제든 우회될 수 있다.
쿼리 파라미터가 조금씩 달라질 수도 있고, 헤더가 달라져 캐시 키가 분산될 수도 있고, TTL이 만료되는 시점에 요청이 몰릴 수도 있다.
배포나 설정 변경으로 캐시가 한 번에 비워질 수도 있다.

그래서 public API는 캐시가 실패했을 때의 원본 서버 요청수를 같이 봐야 한다.

CDN 캐시를 적용했으니 원본 서버는 약해도 된다는 식으로 생각하면 위험하다.
평소 요청의 대부분은 CDN에서 끝나더라도, 캐시 미스나 캐시 무효화 상황에서는 원본 서버가 다시 요청을 받는다.

이를 위해서는 몇 가지 장치가 필요하다.

- Origin 기준 Rate Limit
- 짧은 시간 내 동일 캐시 미스 요청 병합
- TTL 만료 시점 분산
- stale 응답 허용
- 캐시 무효화 범위 최소화
- API별 원본 요청수 모니터링

특히 stale 응답은 public API에서 유용할 때가 많다.

강의 카테고리나 공개 태그 목록이 몇 초, 몇 분 늦게 갱신된다고 해서 큰 문제가 되지 않는다면, 원본 서버 장애 상황에서도 이전 응답을 잠시 더 내려주는 편이 낫다.

항상 최신을 보여주는 것보다, 안정적으로 비슷한 응답을 계속 주는 것이 더 중요한 public API가 있다.

이런 기준이 API마다 정리되어 있으면 캐시 정책을 잡기가 훨씬 편하다.

## Public API의 응답 크기

Public API는 호출 수만큼 응답 크기도 중요하다.

한 번의 응답이 몇 KB 차이 나는 것은 작아 보인다.
하지만 모든 페이지에서 호출되는 public API라면 이야기가 달라진다.

예를 들어 공통 카테고리 API가 모든 페이지에서 호출되고, 응답 크기가 크다면 CDN 캐시를 적용해도 사용자 네트워크 비용과 브라우저 처리 비용은 계속 발생한다.
서버 요청수는 줄었지만 클라이언트가 매번 큰 JSON을 내려받는다면 다른 병목이 생길 수 있다.

그래서 public API는 응답 필드를 private API보다 조금 더 엄격하게 보게 된다.

Private API에서는 화면 개발 편의를 위해 조금 넉넉한 응답을 줄 때도 있다.
어차피 인증된 사용자만 제한된 화면에서 호출하고, 호출량도 상대적으로 예측 가능하기 때문이다.

하지만 public API는 다르다.
호출 주체가 넓고, 호출량이 크고, 캐시되어 여러 사용자에게 반복 전달된다.

응답에 지금 화면에서 쓰지 않는 필드가 계속 포함되어 있다면 그 비용은 계속 누적된다.

Public API의 응답은 작고 안정적일수록 운영하기 쉽다.
필드가 자주 바뀌면 캐시 효율도 떨어지고, 클라이언트 의존성도 커진다.

## 관측 기준도 분리하기

Public API는 모니터링도 별도로 봐야 한다.

전체 API 요청수만 보면 public API의 문제를 놓치기 쉽다.
Public API는 요청량이 많아서 전체 지표를 지배할 수도 있고, 반대로 CDN에서 대부분 처리되면 애플리케이션 서버 지표에서는 거의 보이지 않을 수도 있다.

그래서 최소한 다음 지표는 따로 봐야 한다.

- CDN cache hit ratio
- CDN에서 원본 서버로 전달된 요청수
- public API별 원본 요청수
- public API별 응답 크기
- public API별 p95, p99 응답 시간
- Rate Limit 발생 수
- WAF나 Gateway에서 차단된 요청 수
- 캐시 무효화 횟수와 범위

특히 CDN cache hit ratio만 보면 착시가 생길 수 있다.

Hit ratio가 높아도 전체 트래픽이 크면 원본 요청수는 여전히 많을 수 있다.
반대로 hit ratio가 조금 낮아도 전체 요청량이 작으면 큰 문제가 아닐 수 있다.

그래서 비율과 절대량을 같이 봐야 한다.

```text
cache hit ratio = 95%
total requests = 10,000,000
origin requests = 500,000
```

95%라는 숫자만 보면 좋아 보인다.
하지만 원본 서버로 50만 요청이 들어온다면 그 API가 서버에 주는 부담은 여전히 크다.

캐시율이 높아도 원본 서버 요청수가 많으면 서버 입장에서는 여전히 부담이 크다.
Public API를 볼 때는 비율과 함께 원본 서버가 실제로 맞는 요청수를 같이 봐야 한다.

## Public API를 설계할 때 묻는 질문

Public API를 만들 때는 컨트롤러와 쿼리부터 만들기 전에 몇 가지 질문을 먼저 해보는 것이 좋다.

이 API는 로그인하지 않은 사용자에게도 동일한 응답을 주는가?

사용자, 권한, 쿠키, 지역, 언어, 실험군에 따라 응답이 달라지는가?

CDN에서 캐시할 수 있는가?

캐시할 수 있다면 TTL은 어느 정도까지 허용 가능한가?

최신성이 조금 늦어져도 되는가?

응답 크기는 얼마나 되는가?

이 API는 모든 페이지에서 호출되는가, 특정 화면에서만 호출되는가?

검색 봇이나 스크래퍼가 호출해도 괜찮은가?

비정상 호출이 늘었을 때 어디서 제한할 것인가?

캐시가 비워졌을 때 원본 서버가 견딜 수 있는가?

이 질문들에 답하지 못한 상태에서 public API가 열리면, 문제가 생겼을 때 손댈 수 있는 곳이 애플리케이션 코드밖에 남지 않는다.

Public API는 외부에 공개되어 있다는 이유만으로 단순하지 않다.
오히려 호출자가 불명확하고, 트래픽이 넓게 퍼지고, 캐시와 보안 정책이 함께 필요하기 때문에 더 신중하게 다뤄야 한다.

## 서버가 직접 맞지 않는 구조

Public API의 성능 개선을 서버 내부에서만 찾으면 선택지가 제한된다.

쿼리를 최적화하고,
인덱스를 추가하고,
Redis를 붙이고,
로컬 캐시를 붙인다.

이런 개선은 필요하다.
다만 모두 요청이 서버까지 온 뒤의 개선이다.

Public API에서는 그보다 앞의 계층을 같이 봐야 한다.

이 요청이 정말 애플리케이션 서버까지 와야 하는가?

CDN에서 끝낼 수 있는가?

Gateway나 WAF에서 걸러낼 수 있는가?

Rate Limit으로 원본 서버를 보호할 수 있는가?

응답 크기를 줄여 반복 전달 비용을 낮출 수 있는가?

Public API는 외부 세계와 서비스가 만나는 가장 앞쪽의 인터페이스다.
애플리케이션 서버가 모든 요청을 직접 맞는 구조로 두면, 트래픽이 늘어날수록 비용과 장애 가능성도 같이 커진다.

서버가 더 빠르게 처리하는 것도 필요하다.
하지만 서버가 받지 않아도 되는 요청을 줄일 수 있다면, 그 편이 더 낫다.

요즘 public API를 볼 때는 응답 시간 그래프와 함께 원본 서버 요청수 그래프를 같이 보게 된다.
