# 취준생, 학생분들을 위한 요즘 시대의 AI 코딩 경험 쌓기

학생분들을 대상으로 강연을 갈 때마다 거의 빠지지 않고 받는 질문이 있다.

"AI 코딩 도구를 배우고는 싶은데, 취준생이나 학생 입장에서는 비용이 부담돼요."

"그래도 요즘 개발자들은 다 AI를 쓴다던데, 어떻게든 경험은 쌓아야 하지 않을까요?"

"Cursor, Copilot, Gemini, Claude Code, Codex, Junie 같은 게 너무 많은데 뭘 먼저 써야 하나요?"

질문은 조금씩 다르지만, 안에 들어있는 고민은 비슷하다. AI를 안 쓰자니 뒤처지는 것 같고, 그렇다고 매달 여러 도구를 결제하기에는 부담스럽다. 무료 플랜만 써도 충분한지 모르겠고, 학생 혜택은 어디까지 되는지도 헷갈린다. 막상 도구를 켜도 어디까지 AI에게 맡기고 어디부터는 직접 해야 하는지 애매하다.

이 글은 그 질문에 대한 답을 정리하기 위해 쓴다.

특정 도구를 광고하려는 글은 아니다. 오히려 반대에 가깝다. 학생이나 취업준비생이라면 처음부터 유료 도구를 여러 개 결제할 필요가 없다. 무료 플랜, 학생 혜택, 체험판만 잘 써도 AI 코딩 경험은 충분히 시작할 수 있다. 중요한 것은 "어떤 도구가 제일 좋은가"보다 "내 상황에서 어떤 조합으로 시작하고, 그 도구를 어떤 방식으로 학습에 붙일 것인가"다.

먼저 가격과 플랜 이야기를 하기 전에 전제를 하나 깔고 싶다. 이 글에 나오는 가격, 무료 플랜, 학생 혜택, 사용량 제한은 2026년 5월 작성 시점의 공식 페이지를 참고한 것이다. AI 코딩 도구 쪽은 정책이 정말 자주 바뀐다. 한 달 전에는 무료였던 기능이 유료가 되기도 하고, 학생 플랜 신청이 일시 중단되기도 하고, 반대로 새로운 무료 플랜이 열리기도 한다. 그래서 실제 가입 전에는 반드시 공식 페이지 확인이 필요하다. 이 글에서는 "정확히 몇 달러냐"보다 "어떤 기준으로 고르면 좋으냐"와 "돈을 많이 쓰지 않고도 어떻게 경험을 쌓을 수 있느냐"를 중심으로 이야기하겠다.

다만 무료 도구 목록만 나열하면 반쪽짜리 답이 된다. AI 코딩 도구를 쓰는 목적은 도구 수집이 아니라 코딩 경험을 쌓는 것이기 때문이다. 그래서 도구 이야기를 하기 전에, AI를 학습에 어떻게 붙여야 하는지부터 짚고 가려고 한다.

결론부터 말하면, 요즘 시대의 코딩 학습은 혼자 처음부터 끝까지 외우는 방식도 아니고, AI에게 전부 맡기는 방식도 아니다. 직접 구현해보고, 막히는 지점을 기록하고, AI에게 힌트와 리뷰와 테스트를 받아가며 실무에 가까운 피드백 루프를 빨리 돌리는 방식에 가깝다.

AI는 코드를 대신 제출해주는 대리인이 아니라, 내가 계속 코딩할 수 있게 도와주는 튜터이자 페어 프로그래머이자 코드 리뷰어이자 디버깅 파트너다. 이 차이를 알고 쓰는 사람과 모르는 사람의 차이가 꽤 크게 벌어질 것 같다.

예전에는 코딩을 배울 때 혼자 보내야 하는 시간이 길었다. 문법이 헷갈리면 검색하고, 에러가 나면 스택오버플로우를 뒤지고, 예제 코드를 복사해보고, 안 되면 다시 검색했다. 그 과정 자체가 공부이기도 했다. 삽질을 통해 문법, 도구, 프레임워크, 에러 메시지 읽는 법을 익혔다.

지금은 그 시간이 많이 달라졌다. AI에게 "이 에러가 왜 나는지 설명해줘"라고 물으면 꽤 그럴듯한 원인 후보를 뽑아준다. "이 함수 테스트 케이스 만들어줘"라고 하면 테스트 초안을 만들어준다. "이 코드에서 위험한 부분 리뷰해줘"라고 하면 리뷰어처럼 지적해준다. "이 코드를 더 읽기 좋게 리팩터링하려면 어디부터 보면 좋을까?"라고 물으면 방향을 잡아준다.

그래서 코딩 공부가 쉬워졌냐고 하면, 절반만 맞다.

처음 시작하는 장벽은 낮아졌다. 하지만 그만큼 더 중요한 능력이 생겼다. AI가 만든 코드가 맞는지, 지금 내 상황에 적합한지, 보안상 위험하지 않은지, 테스트로 검증됐는지, 면접에서 내가 설명할 수 있는지 판단하는 능력이다.

AI가 코드를 만들어줘도 그 코드가 왜 동작하는지 모르면 실력이 쌓이지 않는다. 에러 없이 실행돼도 설계가 이상할 수 있고, 지금은 잘 돌아가도 사용자가 조금만 늘면 문제가 터질 수 있다. 인증, 결제, 개인정보, DB 트랜잭션, 권한 처리 같은 부분은 특히 그렇다.

그래서 AI 시대의 초보 개발자는 "코드를 직접 다 외워야 하는 사람"도 아니고, "AI에게 코드를 맡기면 되는 사람"도 아니다. 더 정확히는 AI가 만든 결과물을 읽고, 고치고, 테스트하고, 설명할 수 있는 사람이 되어야 한다.

처음부터 AI에게 "투두 앱 전체 만들어줘"라고 하면 결과물은 빨리 나온다. 하지만 거기서 내가 배운 것은 생각보다 적다. 파일이 왜 이렇게 나뉘었는지, 상태 관리는 왜 이렇게 했는지, API는 왜 이 모양인지, 예외 처리는 어디에 있는지, 테스트는 무엇을 검증하는지 모르면 포트폴리오라기보다 생성물에 가깝다.

반대로 AI를 아예 안 쓰고 혼자만 버티는 것도 좋은 전략은 아니다. 지금 현업 개발자들도 AI를 써서 문서 읽기, 테스트 작성, 코드 리뷰, 로그 분석, 반복 작업 자동화를 하고 있다. 학생과 취준생이 AI 없이 공부해야만 진짜 실력이라는 기준을 세우면, 오히려 실무와 멀어진다.

좋은 기준은 이거다.

먼저 내가 직접 구현해본다. 막히는 지점과 에러를 기록한다. AI에게 정답 코드를 바로 요구하지 않고 힌트, 테스트 케이스, 코드 리뷰, 리팩터링 방향을 요청한다. 작은 기능 단위로 AI 에이전트를 사용한다. 마지막에는 내가 코드를 이해하고, 테스트하고, 디버깅하고, 면접에서 설명할 수 있어야 한다.

처음부터 이 루틴을 추천한다.

1. 20~30분은 직접 구현한다.
2. 막힌 부분을 기록한다.
3. AI에게 정답이 아니라 힌트를 요청한다.
4. 내가 작성한 코드를 리뷰해달라고 한다.
5. 실패할 테스트 케이스를 만들어달라고 한다.
6. 리팩터링 방향을 묻는다.
7. 마지막으로 면접에서 설명할 수 있게 요약한다.

이 루틴에서 제일 중요한 것은 1번과 2번이다. 바로 AI를 켜면 내가 어디서 막히는지 모른다. 막히는 지점을 모르면 공부할 지점도 모른다. "회원가입 API에서 비밀번호 암호화 후 저장까지는 했는데, 중복 이메일 검증을 어디서 해야 할지 모르겠다" 정도로 막힌 지점을 말할 수 있어야 AI도 튜터처럼 도와줄 수 있다.

프롬프트도 이렇게 바꾸면 좋다.

```text
나는 취업 준비생이고, 이 코드는 내가 직접 먼저 작성했다.
정답 코드를 바로 새로 작성하지 말고 다음 순서로 도와줘.

1. 현재 코드에서 가장 큰 문제 3개를 설명해줘.
2. 내가 직접 고칠 수 있도록 힌트만 먼저 줘.
3. 실패할 만한 테스트 케이스 5개를 만들어줘.
4. 내가 수정한 뒤에는 코드 리뷰어처럼 리뷰해줘.
5. 마지막에 시간복잡도, 공간복잡도, 면접에서 설명할 포인트를 정리해줘.
```

이 프롬프트의 핵심은 "정답 코드를 바로 새로 작성하지 말라"는 부분이다. AI는 친절해서 바로 답을 주려고 한다. 그런데 학습에서는 바로 답을 받는 게 항상 좋은 게 아니다. 특히 코딩테스트, 알고리즘, 기본 CRUD, 인증 흐름, DB 설계처럼 실력을 쌓아야 하는 영역에서는 힌트와 테스트와 리뷰를 받는 쪽이 훨씬 낫다.

예를 들어 코딩테스트 문제를 풀 때는 처음 30분은 AI를 꺼두는 것도 좋다. 종이에 접근법을 적어보고, 시간복잡도를 예상하고, 직접 구현해본다. 그 다음 AI에게 "내 접근법의 반례를 찾아줘"라고 한다. "정답 코드 알려줘"가 아니라 "이 풀이가 왜 틀릴 수 있는지 알려줘"라고 하는 것이다. 이러면 AI는 답안지가 아니라 스파링 상대가 된다.

프로젝트에서도 비슷하다. "블로그 서비스 만들어줘"라고 하지 말고 기능을 작게 쪼갠다.

회원가입 요청 DTO를 설계한다. 이메일 중복 검증 테스트를 만든다. 게시글 생성 API를 만든다. 페이지네이션 쿼리를 작성한다. 파일 업로드 저장 경로를 정한다. 배포 환경에서 환경변수를 분리한다. 이런 작은 단위로 AI를 쓰면 내가 프로젝트의 주도권을 잃지 않는다.

AI 코딩 에이전트는 여기서 한 단계 더 들어간다. 단순 채팅형 AI는 설명과 코드 조각을 잘 준다. 에이전트형 도구는 프로젝트 폴더를 읽고, 여러 파일을 고치고, 테스트를 실행하고, 터미널 명령까지 제안하거나 실행한다. ChatGPT Codex, Claude Code, Gemini CLI, Cursor Agent, Windsurf Cascade, JetBrains Junie, Replit Agent, GitHub Copilot agent mode 같은 도구들이 여기에 가깝다.

에이전트는 잘 쓰면 강력하다. 하지만 처음부터 너무 큰 일을 맡기면 학습에는 별로 좋지 않다. "전체 프로젝트 만들어줘"보다 "게시글 수정 API에 대한 실패 테스트를 먼저 추가하고, 현재 구현이 왜 실패하는지 설명해줘"가 낫다. "전체 리팩터링해줘"보다 "이 컨트롤러에서 서비스로 옮겨야 할 책임을 3개만 찾아줘"가 낫다.

작은 기능 단위로 맡기고, 변경 diff를 읽고, 테스트를 돌리고, 내가 이해한 내용을 README나 커밋 메시지로 정리하는 흐름이 좋다. 이게 실무에 훨씬 가깝다. 실무에서도 좋은 개발자는 도구에게 일을 통째로 던지는 사람이 아니라, 일을 잘게 나누고, 결과를 검증하고, 위험한 지점을 판단하는 사람이다.

도구를 고를 때는 가격만 보면 안 된다. 무료 플랜이 충분한지, 학생 혜택이 있는지, IDE 안에서 자연스럽게 쓸 수 있는지, 터미널에서 쓸 수 있는지, 프로젝트 전체를 이해하는지, 코드 생성보다 코드 리뷰와 디버깅에 좋은지, 포트폴리오 프로젝트를 끝까지 완성하는 데 도움이 되는지 봐야 한다.

한 번에 비교하면 대략 이렇다. 가격과 플랜은 변동될 수 있으므로 공식 페이지 확인이 필요하다.

내가 학생이나 취준생이라면 가장 먼저 Gemini Code Assist와 Gemini CLI부터 열어볼 것 같다. 이유는 단순하다. 무료로 시작하기 쉽고, 사용량이 비교적 넉넉하며, VS Code나 JetBrains IDE에서도 쓸 수 있고, 터미널에서도 쓸 수 있다. 처음 AI 코딩 도구를 배울 때는 유료 도구를 고르기 전에 "AI에게 어떤 질문을 해야 도움이 되는지"부터 익혀야 한다. 이 연습에는 Gemini가 좋은 출발점이 될 수 있다.

그 다음에는 내가 어떤 개발 환경에 익숙한지에 따라 갈린다.

VS Code와 GitHub 중심으로 공부한다면 GitHub Copilot을 붙이는 것이 자연스럽다. 자동완성, 코드 설명, 간단한 채팅, GitHub 연동이 좋다. 이미 GitHub Student Developer Pack을 받을 수 있다면 Copilot Student도 확인해볼 만하다. 다만 2026년 5월 기준으로도 Copilot 플랜 정책은 계속 바뀌고, 신규 가입이나 학생 플랜 신청 가능 여부가 일시적으로 달라질 수 있으니 공식 페이지를 확인해야 한다.

프로젝트를 AI IDE와 함께 만들어보고 싶다면 Cursor나 Windsurf가 좋다. 둘 다 "AI와 같이 프로젝트를 만든다"는 느낌이 강하다. 여러 파일을 읽고, 수정하고, 리팩터링하고, 테스트를 추가하는 흐름이 자연스럽다. 프론트엔드나 풀스택 포트폴리오를 만들 때 특히 체감이 크다. 대신 이 도구들은 편한 만큼 위험도 있다. 내가 모르는 파일을 AI가 많이 고치면 학습이 아니라 구경이 된다. 그래서 Cursor나 Windsurf를 쓸 때는 반드시 기능 단위를 작게 잡고, 변경 diff를 읽고, 커밋을 작게 나눠야 한다.

ChatGPT를 이미 공부에 많이 쓰고 있다면 Codex 계열을 같이 보는 것도 좋다. 설명, 코드 리뷰, 문서화, README 작성, 면접 답변 정리까지 한곳에서 이어가기 좋다. 로컬 Codex 앱이나 CLI를 쓰면 프로젝트 폴더를 기반으로 작업할 수 있고, 클라우드 작업까지 연결되는 경우도 있다. 특히 "왜 이렇게 구현했는지 설명해줘", "면접에서 이 프로젝트를 어떻게 말하면 좋을까?", "README를 포트폴리오용으로 정리해줘" 같은 작업에 잘 맞는다.

터미널에서 작업하는 데 익숙하고, 백엔드나 시스템 개발 쪽을 공부한다면 Claude Code도 좋은 선택지다. 긴 로그를 읽고, 실패한 테스트 원인을 추적하고, 코드베이스를 탐색하는 경험이 좋다. 다만 터미널 기반 도구는 파일 변경과 명령 실행 권한을 더 신중하게 봐야 한다. 초보자라면 처음부터 자동 실행 모드를 넓게 열기보다, 읽기와 제안 중심으로 시작하는 편이 낫다.

설치 자체가 어렵거나, 학교 컴퓨터나 태블릿, 크롬북처럼 로컬 개발환경을 마음대로 만들기 어려운 상황이라면 Replit Agent가 유용하다. 브라우저에서 만들고, 실행하고, 배포까지 해볼 수 있다. 초보자에게 "내가 만든 것이 URL로 열린다"는 경험은 꽤 중요하다. 다만 Replit만 쓰면 로컬 개발환경, Git, 터미널, 배포 인프라를 깊게 배우기 어렵다. 처음 한두 개 프로젝트를 끝까지 완성하는 데 쓰고, 이후에는 로컬 환경도 같이 익히는 편이 좋다.

AWS, Java, Spring, 클라우드 백엔드를 목표로 한다면 Amazon Q Developer도 볼 만하다. 일반적인 AI 코딩 도구라기보다 AWS 맥락에서 강점이 있다. AWS 서비스 설명, 클라우드 백엔드 작업, Java 애플리케이션 현대화 같은 흐름에 잘 맞는다. Java/Spring 취업 준비생이라면 IntelliJ IDEA와 함께 쓰면서 "로컬 백엔드 구현 + AWS 배포 + 클라우드 운영"까지 이어보는 식으로 활용할 수 있다.

그리고 학생이라면 JetBrains Student Pack은 정말 한 번은 신청해볼 만하다. Java/Spring 백엔드를 공부한다면 IntelliJ IDEA Ultimate 경험은 아직도 크다. 프로젝트 탐색, 리팩터링, 테스트 실행, DB 도구, HTTP Client, Spring 지원 같은 것들이 학습 속도를 꽤 올려준다. Python이면 PyCharm, 웹이면 WebStorm, DB 쪽이면 DataGrip도 좋다. 실무에서 많이 쓰는 전문 IDE를 학생 때 무료로 익힐 수 있다면 굳이 놓칠 이유가 없다.

다만 여기서 많이 헷갈리는 부분이 있다. JetBrains Student Pack으로 IntelliJ IDEA Ultimate를 무료로 쓴다고 해서 Junie나 JetBrains AI를 충분히 무료로 쓰는 것은 아니다. Junie는 JetBrains AI 서비스와 크레딧을 사용한다. AI Trial, AI Free, AI Pro, AI Ultimate 같은 별도 플랜과 크레딧 정책이 있다. 그래서 "학생이라 IntelliJ Ultimate는 무료"와 "JetBrains AI/Junie를 충분히 무료로 쓴다"는 다른 이야기다. 이 둘을 분리해서 봐야 한다.

| 상급 | 테스트가 포함된 백엔드 API, CI/CD, Docker, DB 마이그레이션, OAuth 로그인, 관리자 페이지, 성능 개선 리포트 | 리팩터링 방향, 보안 체크리스트, 성능 병목 후보, CI 실패 로그 분석, 면접 설명 포인트 |

각 프로젝트에서 AI에게 맡기면 좋은 일은 분명히 있다.

요구사항을 정리하게 한다. DB 스키마 초안을 만들게 한다. 테스트 케이스를 제안하게 한다. 에러 로그를 분석하게 한다. 코드 리뷰를 받는다. README 초안을 만든다. 면접에서 설명할 포인트를 정리한다.

반대로 맡기면 안 되는 일도 있다.

전체를 무작정 대신 만들게 하지 않는 것. 내가 이해하지 못한 코드를 그대로 포트폴리오에 넣지 않는 것. 코딩테스트 풀이를 그대로 베끼지 않는 것. 인증, 결제, 개인정보, 권한, DB 트랜잭션 같은 위험한 코드를 검토 없이 넣지 않는 것.

"내가 설명하지 못하는 코드는 내 코드가 아니다"라는 기준을 두면 좋다.

물론 현실적으로 모든 코드를 100% 다 설명하기는 어렵다. 라이브러리 내부 구현까지 외울 필요도 없다. 하지만 적어도 내가 작성한 API 흐름, 데이터 모델, 예외 처리, 테스트 전략, 배포 방식, AI가 고쳐준 주요 변경점은 설명할 수 있어야 한다. 면접에서 "이 부분은 AI가 제안했고, 제가 테스트를 추가해서 이렇게 검증했습니다"라고 말할 수 있으면 오히려 좋다. AI를 썼다는 사실보다, AI 결과물을 어떻게 검증했는지가 중요하다.

그래서 포트폴리오 프로젝트를 만들 때는 README에 이런 내용을 남기면 좋다.

- 직접 구현한 핵심 기능
- AI에게 도움받은 부분
- AI 제안을 그대로 쓰지 않고 수정한 부분
- 실패했던 버그와 해결 과정
- 테스트로 검증한 내용
- 배포하면서 겪은 문제
- 다음에 개선하고 싶은 부분

이런 기록이 있으면 프로젝트가 훨씬 진짜 같아진다. 취업 준비생의 포트폴리오에서 가장 아쉬운 것은 기능 수가 적은 것이 아니라, 본인이 무엇을 고민했는지 보이지 않는 경우다. AI 시대에는 결과물만 보면 누가 얼마나 이해했는지 더 알기 어렵다. 그래서 과정 기록이 더 중요해진다.

코딩테스트 연습은 별도 규칙을 두는 것이 좋다. 매일 1문제를 푼다면 처음 30분은 AI 금지 시간으로 둔다. 접근법을 먼저 쓰고, 직접 구현하고, 예제 입출력을 통과시켜본다. 그 다음 AI에게 힌트, 반례, 시간복잡도 리뷰를 요청한다. 마지막에 정답 풀이를 봤다면 내 풀이와 다른 점을 정리한다. 이렇게 하면 AI가 사고 과정을 빼앗아가지 않고, 오히려 사고 과정을 훈련시켜준다.

실무형 프로젝트에서는 커밋 단위가 중요하다. AI IDE나 에이전트가 여러 파일을 한 번에 고치더라도, 내가 커밋할 때는 기능 단위로 나눠야 한다. 예를 들어 "회원가입 기능 추가", "회원가입 검증 테스트 추가", "비밀번호 암호화 적용", "README 사용법 추가"는 서로 다른 목적이다. 학습에서도 커밋 단위를 나누면 내가 무엇을 했는지 정리하기 좋아지고, 나중에 면접에서 설명하기도 쉬워진다.

도구를 고르는 순서는 이렇게 추천한다.

먼저 무료 도구로 시작한다. Gemini Code Assist, Gemini CLI, GitHub Copilot Free, ChatGPT 무료 버전, Replit Starter 정도로 충분하다. 학생이면 GitHub Student Developer Pack과 JetBrains Student Pack을 신청한다. 여기까지는 돈보다 귀찮음의 문제에 가깝다.

그 다음 한 가지 프로젝트를 정한다. 예를 들어 "로그인/회원가입/게시판/검색/배포가 있는 작은 웹앱" 정도면 좋다. 프로젝트를 정했으면 AI 도구를 1~2개만 고른다. 프론트엔드 중심이면 Cursor나 Windsurf, Java/Spring이면 IntelliJ IDEA Ultimate와 Gemini 또는 Junie trial, 설명과 리뷰 중심이면 ChatGPT/Codex, 터미널 디버깅 중심이면 Claude Code나 Gemini CLI를 선택한다.

그리고 2주 동안 하나의 프로젝트를 끝까지 간다. 도구를 비교하느라 프로젝트를 바꾸지 않는다. AI가 만든 코드를 읽고, 테스트하고, README를 쓰고, 배포까지 해본다. 완성도가 조금 낮아도 괜찮다. 끝까지 가본 프로젝트 하나가, 시작만 한 프로젝트 다섯 개보다 낫다.

바로 실천할 수 있는 2주 루틴을 적어보면 이렇다.

이 2주 루틴에서 중요한 것은 매일 새로운 도구를 써보는 것이 아니다. 같은 프로젝트에서 AI를 튜터, 리뷰어, 디버깅 파트너로 계속 써보는 것이다. "오늘은 Gemini에게 에러를 물어봤다", "내일은 Cursor로 전체를 다시 만들었다", "모레는 Claude Code로 갈아탔다"처럼 움직이면 경험이 흩어진다. 한 프로젝트를 기준으로 도구를 붙여야 비교도 되고 학습도 된다.

AI 시대의 코딩 공부에서 제일 위험한 착각은 "이제 구현력은 덜 중요해졌다"는 생각이다. 오히려 구현력의 기준이 바뀌었다. 예전에는 빈 파일에서 문법을 얼마나 잘 떠올리느냐가 중요했다면, 이제는 이미 만들어진 코드와 AI가 제안한 코드를 얼마나 잘 읽고 판단하고 고치느냐가 중요해졌다.

실무에서는 늘 남이 쓴 코드를 읽는다. 예전 팀원이 만든 코드, 오픈소스 코드, 라이브러리 코드, 문서와 다른 코드, 급하게 만든 코드, AI가 만든 코드까지 읽어야 한다. 그러니 AI 시대의 구현력은 "혼자 타이핑하는 능력"만이 아니다. 읽기, 검증하기, 나누기, 테스트하기, 설명하기까지 포함된다.

취업 준비생과 학생분들에게는 오히려 좋은 시대일 수 있다. 예전에는 좋은 멘토를 만나거나 좋은 팀 프로젝트를 경험해야만 받을 수 있던 피드백을, 지금은 어느 정도 혼자서도 만들 수 있다. 물론 AI가 완벽한 멘토는 아니다. 틀린 말을 자신 있게 하기도 하고, 프로젝트 맥락을 놓치기도 하고, 보안상 위험한 코드를 제안하기도 한다. 그래서 더더욱 내가 판단해야 한다.

AI가 코드를 대신 써주는 시대라기보다, 더 빨리 배우는 시대에 가깝다. 같은 에러에서 하루를 보내던 시간을 한 시간으로 줄이고, 혼자 놓치던 반례를 더 빨리 발견하고, README와 테스트와 배포까지 경험해볼 수 있는 시대다.

다만 그 속도가 실력으로 남으려면 마지막에는 항상 내 말로 정리해야 한다.

"이 코드는 왜 이렇게 만들었나?"

"어떤 대안을 검토했나?"

"AI가 제안한 것 중 무엇을 버렸나?"

"테스트로 무엇을 확인했나?"

"다시 만든다면 무엇을 바꿀 건가?"

이 질문에 답할 수 있다면 AI를 잘 쓰고 있는 것이다. 답할 수 없다면 아직 내 코드가 아니다.

처음부터 모든 도구를 잘 쓸 필요는 없다. 예산이 없다면 Gemini부터 시작해도 좋다. 학생이라면 JetBrains Student Pack과 GitHub Student Developer Pack을 신청해도 좋다. 프론트엔드 포트폴리오라면 Cursor나 Windsurf 무료 플랜을 체험해도 좋다. Java/Spring이라면 IntelliJ IDEA Ultimate와 Gemini, 필요하면 Junie trial을 써봐도 좋다. 배포가 막막하면 Replit Agent로 첫 URL을 만들어봐도 좋다.

중요한 것은 도구의 이름이 아니라 루틴이다.

직접 해본다. 막힌다. 기록한다. 힌트를 받는다. 테스트를 만든다. 리뷰를 받는다. 고친다. 배포한다. 설명한다.

이 루틴을 2주만 제대로 돌려도, AI를 그냥 "코드 뽑는 기계"로 쓰는 사람과는 전혀 다른 경험이 쌓인다. 그리고 그 경험이 결국 포트폴리오와 면접에서 드러난다.
